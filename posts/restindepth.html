What does REST really mean?

a server has a resource and a client requests a representation of the resource’s state (data)

the client doesn’t care about the server’s internal implementations at all, simply cares about transferring representational states

rest architecture makes it simpler for systems to communicate on the web, as just HTTP calls are utilized (rather than SOAP, for example)

specifically, HTTP requests, responses, methods, and status codes are utilized!

6 REST Constraints

App will exhibit many desirable properties (more in-depth info below) if all REST constraints are followed

  1. all communication between server and client should be stateless. so all client session info is maintained by the client, however this also means each request must contain all necessary info, no context information is available
  2. server responses must be marked as cacheable or non-cacheable (performance)
  3. uniform interface - four part constraint
    • resources are identified via URIs (remember, the resources themselves are different from the representations)
    • if a client holds a representation and any necessary metadata, it has enough info to modify the resource on the server (with appropriate permissions)
    • each message must contain all info needed to inform the server on processing it (self-descriptive messages)
    • hypermedia as the engine of application state, or HATEOAS, allows the client to have minimal knowledge about HOW to interact with the server. this is because the server dynamically generates responses to the client using hypermedia (allows sending data and specifying actions)
  4. it’s best to symbolize a resource with a webpage, considering each resource should have one logical UI
  5. client should only know immediate layer it is communicating with. should not be able to tell if it is communicating with intermediary or server itself
  6. code on-demand is actually an optional constraint in rest arch. it means the server can send code the client should execute at runtime to extend functionality. e.g. javascript in web pages

Desirable Properties Achieved with REST

  • lightweight and flexible, allows many data formats
  • simplicity, it just acts as an extension of HTTP
  • highly scalable
  • easy to implement, allows for clear separation of front end and back end
  • caching is easier, this is critical for application performance

These qualities are quite essential in networked applications!
